# lk-seller

Веб-приложение — личный кабинет продавца с интегрированным AI-ассистентом, который помогает улучшать описания объявлений. Продавец видит список своих объявлений, выбирает нужное и может просмотреть карточку товара, перейти к редактированию и запросить рекомендации от нейросети. Сервис анализирует содержимое карточки товара и предлагает улучшения текста.

## Требования

- **Node.js** v20 или новее (`nvm use` подхватит версию из `.nvmrc`).

## Установка и запуск Front-end

```bash
cd ./frontend/
npm install
npm run dev
```

Откройте в браузере адрес `http://localhost:5173`.

## Установка и запуск Back-end

```bash
cd ./server/
npm install
port=8080 npm start
```

## Установка и запуск Ollama

1. Установите Ollama

Загрузите модель:

```
ollama pull llama3
```

Убедитесь, что Ollama запущена:

```
ollama serve
```

## Ход решения и принятые решения

Если в требованиях оставалась неопределенность, были приняты следующие решения:

1. **Контракт списка объявлений расширен на backend**
   - В `GET /items` добавлены поля `id` и `createdAt` в каждом элементе списка.
   - **Почему:** без `id` невозможно надежно строить переход на карточку (`/ads/:id`), а `createdAt` нужен для корректной сортировки по новизне.

2. **Фильтрация и пагинация выполняются через API-параметры**
   - Используются `q`, `categories`, `needsRevision`, `limit`, `skip`.
   - **Почему:** это соответствует требованиям и держит UI простым, не раздувая клиентскую логику.

3. **Сортировка по новизне строится по `createdAt`**
   - На фронтенде "по новизне" опирается на дату создания, а не на косвенные признаки (например, `id`).
   - **Почему:** это предсказуемо и корректно с точки зрения бизнес-смысла.

4. **Защита от устаревшего ответа backend**
   - На фронтенде добавлена безопасная обработка кейса, когда у элемента списка нет `id` (элемент не кликабелен + предупреждение).
   - **Почему:** пока локально поднята старая версия сервера, интерфейс не должен ломаться.

5. **Текущее хранилище данных - in-memory из `items.json`**
   - Изменения применяются в процессе работы сервера и не пишутся обратно в файл.
   - **Почему:** для тестового задания это упрощает реализацию, ускоряет разработку и достаточно для проверки сценариев.

6. **AI-часть вынесена на следующий этап**
   - Сначала реализована стабильная CRUD/UX-основа кабинета продавца.
   - **Почему:** так проще обеспечить качество архитектуры и избежать смешивания нескольких сложных задач в один нестабильный шаг.

7. **Исправлен CORS preflight для операций сохранения**
   - На backend добавлена обработка `OPTIONS`-запросов с ответом `204`, а также заголовки:
     - `Access-Control-Allow-Methods: GET,PUT,POST,PATCH,DELETE,OPTIONS`
     - `Access-Control-Allow-Headers: Content-Type, Authorization`
   - **Почему:** браузер перед `PUT/PATCH` выполняет preflight (`OPTIONS`). Без корректного ответа preflight запрос на сохранение блокировался, и изменения из формы редактирования не отправлялись.

8. **Зафиксирован рабочий контракт для редактирования**
   - `PUT /items/:id` используется как основной способ сохранения карточки объявления с возможностью смены категории, цены, описания и параметров.
   - **Почему:** это дает единый и предсказуемый поток сохранения между страницей просмотра, редактирования и списком объявлений.
